## 一、角色定义

| 角色 | 代号 | 描述 |
|------|------|------|
| 客户 | CU | 信用卡持卡人，通过 Web/App 咨询信用卡相关问题 |
| 坐席 | AG | 银行客服代表，在 Agent Workbench 接听客户来电 |
| 运营 | OP | 知识库管理员，上传/维护知识文档和质检规则 |
| 系统 | SYS | SmartCS 平台自动行为，无需人工介入 |

## 二、会话状态机

会话是贯穿整个服务的核心实体，遵循以下生命周期：

```
BOT ──→ HANDOFF ──→ ASSIST ──→ ENDED
 │                   ↑         │
 └───────────────────┘         │
  (客户回退到自助)              │
                               ↑
              (坐席/客户主动结束)
```

| 阶段 | 触发条件 | 系统行为 |
|------|---------|---------|
| **BOT** | 客户发起会话 | LangGraph Agent 处理消息，意图分类+RAG+LLM 生成 |
| **HANDOFF** | L1/L2/L3 转人工触发 或 挂失/投诉意图 | 调用 star-connection 创建人工会话，返回 pollUrl |
| **ASSIST** | star-connection 回调 session/update | 坐席 WS 注册，OE 编排启动辅助推送 |
| **ENDED** | star-connection 回调会话结束 | WS 清理，反馈归档，审计日志写入 |

## 三、核心交互流程

```
                                          │
                              ┌───────────┴───────────┐
                              │   LangGraph Agent     │
                              │                       │
                              │  1. 意图分类           │
                              │     Rule 快路 (regex)  │
                              │     LLM 慢路 (Qwen2.5) │
                              │                       │
                              │  2. 路由分发           │
                              │     knowledge /        │
                              │     business /         │
                              │     fallback           │
                              │                       │
                              │  3. 分支处理           │
                              │     RAG检索→LLM生成    │
                              │     模板→直转人工      │
                              │     问候/引导          │
                              │                       │

```

```
                              │  ├─────────────────┤  │
                              │  │ D2 营销评估      │  │
                              │  │ 情绪+意图+Suppress│ │
                              │  │ 冷却5轮           │  │
                              │  ├─────────────────┤  │
                              │  │ D3 风控评估      │  │

                              │  └────────┬────────┘  │
                              │           │           │

                              │  └────────┬────────┘  │
                              │           │           │
                              │  WAITING_RESULTS → COMPLETED │

                              │  · 优先级融合(BLOCK>WARN>PASS)│

                              WS 推送到坐席 ←──────────┘
```

### 流程 3：知识库入库与检索

```
OP ──→ POST /kb/documents ──→ 校验格式 ──→ MinIO 存储
                                        │
                    ┌───────────────────┴───────────────────┐
                    │           入库管线                      │
                    │                                       │
                    │  解析 → 清洗 → 分块 → 嵌入 → 双写 → 发布 │
                    │  PDF     去页眉  FAQ对   Ollama  ES      Kafka │
                    │  DOCX    控制符  层级块  TEI     Milvus        │
                    │  HTML    去重    表格保护  batch=128              │
                    │  MD/XLSX         列表保持                      │
                    └───────────────────────────────────────┘

                    ┌───────────────────┬───────────────────┐
                    │           混合检索                      │
                    │                                       │
                    │  查询嵌入 → Milvus 向量检索 (top_k×3)   │
                    │  查询文本 → ES BM25 检索 (top_k×3)      │
                    │  RRF 融合: score = Σ(1/(k+rank_i))     │
                    │  可选 Reranker 精排                      │
                    │  置信度阈值过滤 → 截断 top_k              │
                    │  Parent-Child 扩展: 命中子块→附加父块    │
                    └───────────────────────────────────────┘

降级矩阵:
  ES ✅ Milvus ✅ → 混合检索 (最优)
  ES ✅ Milvus ❌ → BM25 Only
  ES ❌ Milvus ✅ → 向量 Only
  ES ❌ Milvus ❌ → 空结果
  Embedding 熔断  → 自动降级 BM25
  Reranker 失败   → 使用 RRF 原始排序
```

## 四、用户故事

### US-1：客户账单查询

> **作为** 客户，**我想要** 咨询信用卡账单问题，**以便** 快速了解我的账单详情，不需要等待人工客服。

| # | 角色 | 交互/动作 | 系统行为 |
|---|------|----------|---------|
| 1 | CU | 输入"我上个月账单多少钱" | POST /chat/send → Redis LPUSH → 返回 accepted |
| 2 | SYS | — | Worker BRPOP 消费 |
| 3 | SYS | — | Rule 快路匹配 `bill_query` (conf=0.85) |
| 4 | SYS | — | 路由到 knowledge_agent → RAG 混合检索 (ES+Milvus) |
| 5 | SYS | — | LLM 基于检索上下文生成回答 (source=llm) |
| 6 | SYS | — | TransferChecker: L1 未触发, L2 未触发, L3 未触发 |
| 7 | SYS | — | 结果写入 Redis, 长轮询返回 PollResponse |
| 8 | CU | 看到账单查询结果 | — |

**验收标准**:
- [ ] 客户发送消息后 5s 内收到回复
- [ ] 回复内容基于知识库检索，不编造信息
- [ ] 回复附带来源标识 (llm/retrieval/template)
- [ ] 会话历史正确记录本轮对话

**降级路径**:
- LLM 不可用 → 检索摘要 (source=retrieval)
- LLM + 检索都不可用 → 意图模板 (source=template, 提示"输入转人工")
- Embedding 不可用 → BM25 Only 检索

### US-2：客户关键词转人工

> **作为** 客户，**我想要** 直接和真人客服沟通，**以便** 我的问题能得到更专业的处理。

| # | 角色 | 交互/动作 | 系统行为 |
|---|------|----------|---------|
| 1 | CU | 输入"我要转人工" | POST /chat/send → 入队 |
| 2 | SYS | — | Rule 快路匹配 `transfer_agent` (conf=0.95) |
| 3 | SYS | — | 路由到 business_agent → should_transfer=True |
| 4 | SYS | — | TransferChecker: L1 关键词"转人工"命中 |
| 5 | SYS | — | handoff_node: 会话阶段 → HANDOFF |
| 6 | SYS | — | star_client.create_session() → 获取 pollUrl |
| 7 | CU | 看到"正在为您转接人工客服"+ 跳转链接 | — |

**验收标准**:
- [ ] L1 关键词 ("人工"/"转人工"/"真人"/"投诉") 立即触发转接
- [ ] 会话阶段正确过渡到 HANDOFF
- [ ] star-connection 返回有效的 pollUrl
- [ ] star-connection 不可用时，提示"人工客服系统暂不可用"

**待讨论**:
- L1 关键词列表是否需要支持运营后台动态配置？
- star-connection 不可用时的重试策略？

### US-3：客户累积低置信度自动转人工

> **作为** 客户，**我想要** 在反复得不到满意回答时自动转接人工，**以便** 不必手动请求转接。

| # | 角色 | 交互/动作 | 系统行为 |
|---|------|----------|---------|
| 1-3 | CU | 连续 3 轮提问，每轮 Bot 回答置信度低于阈值 | low_confidence_streak 递增到 3 |
| 4 | CU | 输入第 4 个问题 | Worker 消费 |
| 5 | SYS | — | TransferChecker: L1 未命中, L2 未命中, **L3 命中** (streak ≥ 3) |
| 6 | SYS | — | handoff_node → 会话转人工 |

**验收标准**:
- [ ] 连续 low_confidence_streak 达到阈值 (默认 3) 时自动转接
- [ ] 客户不需要手动输入"转人工"
- [ ] 低置信度定义: primary_confidence < 配置阈值
- [ ] 转接前 Bot 发出提示"正在为您转接更专业的客服"

**待讨论**:
- L3 阈值 (当前 3 轮) 是否合理？是否需要按意图类型区分？
- 是否需要"3/5 轮中有 N 轮低置信"的更宽松判定？

### US-4：坐席接收实时辅助推送

> **作为** 坐席，**我想要** 在接听客户来电时实时收到话术推荐、知识片段和合规提醒，**以便** 更专业高效地服务客户。

| # | 角色 | 交互/动作 | 系统行为 |
|---|------|----------|---------|
| 1 | AG | 在 Workbench 打开会话 | WS /ws/{session_id} → 注册连接池 → 收到 assist_ready |
| 2 | CU | 说"我想提升信用卡额度" | star-connection → POST /analyze |
| 3 | SYS | — | 意图分类 → limit_query (conf=0.82) |
| 4 | SYS | — | OE EVALUATING: D1 激活, D2 被 suppress (D1 激活→压制营销), D3 始终激活 |
| 5 | SYS | — | OE DISPATCHING: E1∥E3 并行 |
| 6 | SYS | — | E1: LangGraph DAG → 服务卡片 (话术+知识) |
| 7 | SYS | — | E3: AlertEngine → PASS |
| 8 | SYS | — | 仲裁: PASS → 服务卡片, 营销隐藏 (D2 被压制) |
| 9 | SYS | — | PII 脱敏 → CAS 写回 → WS 推送 |
| 10 | AG | 看到提额话术 + 相关知识片段 | — |

**验收标准**:
- [ ] 从客户发言到坐席收到推送 < 3s (正常情况)
- [ ] 推送内容经过 PII 脱敏
- [ ] 服务场景下营销被自动压制
- [ ] WebSocket 断连后自动清理连接池
- [ ] 心跳 15s 间隔, 30s 超时断开

**待讨论**:
- 推送延迟 3s 的 SLA 是否满足实时性要求？
- 坐席端是否需要"暂停推送"按钮？
- 节流窗口 800ms 是否需要可配置？

### US-5：坐席遇到合规风险告警

> **作为** 坐席，**我想要** 在客户对话中出现合规风险时收到实时警告，**以便** 避免不当承诺造成监管风险。

| # | 角色 | 交互/动作 | 系统行为 |
|---|------|----------|---------|
| 1 | CU | 说"你们能不能保证帮我提额" | POST /analyze |
| 2 | SYS | — | D3 始终激活 → E3 执行 |
| 3 | SYS | — | E3: AlertEngine 检测到"保证" → WARN 级别 |
| 4 | SYS | — | 仲裁: risk_action=WARN → 服务卡片 + 风险徽章 + 营销缩小 |
| 5 | SYS | — | WS 推送 (关键告警绕过节流) |
| 6 | AG | 看到风险徽章"避免过度承诺" | — |

**极端场景**: 客户说"帮我套现" → E3 BLOCK → 仅显示风险卡片, 营销完全隐藏

**验收标准**:
- [ ] 6 条内置合规规则正常触发 (欺诈承诺/身份证号/过度承诺/脏话/手机号/敏感信息)
- [ ] critical 级别告警 → BLOCK, warning 级别 → WARN
- [ ] BLOCK 时营销卡片完全隐藏
- [ ] WARN 时营销卡片降级为小卡片
- [ ] 关键告警绕过节流机制
- [ ] 合规短语 ("保证收益"/"稳赚不赔"等) 被过滤为 [已过滤]

**待讨论**:
- 是否需要坐席端对告警进行确认/消音操作？
- BLOCK 后坐席是否需要手动解锁才能继续对话？
- 合规规则是否需要支持运营后台动态增删？

### US-6：坐席提供隐式反馈

> **作为** 坐席，**我想要** 对系统推送的建议给出反馈（采纳/修改/拒绝），**以便** 系统后续能提供更精准的建议。

| # | 角色 | 交互/动作 | 系统行为 |
|---|------|----------|---------|
| 1a | AG | 直接使用推送话术 | POST /feedback (action=accept, confidence=1.0) |
| 1b | AG | 修改后使用 | POST /feedback (action=modify, confidence=0.5, modify_fields=[...]) |
| 1c | AG | 忽略推送 | POST /feedback (action=reject, confidence=0.0) |
| 2 | SYS | — | 反馈缓冲 3 秒 (允许撤销) |
| 3a | AG | 3 秒内点击撤销 | POST /feedback/undo → 缓冲区移除 |
| 3b | SYS | 3 秒到期 | CAS 写入 Redis → 反馈日志归档 |

**置信度映射**:

| 动作 | 置信度 | 含义 |
|------|--------|------|
| accept | 1.0 | 完全采纳 |
| modify | 0.5 | 部分采纳 |
| partial_accept | 0.3 | 复制部分内容 |
| reject | 0.0 | 完全忽略 |

**验收标准**:
- [ ] 4 种反馈动作正确映射置信度
- [ ] 3 秒内可撤销 (undo)
- [ ] 超过 3 秒不可撤销
- [ ] 反馈通过 CAS 写入 Redis，不丢失
- [ ] 并发反馈不冲突

**待讨论**:
- 3 秒缓冲窗口是否合理？
- 反馈数据当前未用于模型优化，后续是否需要构建反馈→训练闭环？
- 是否需要显式反馈（坐席主动评分）补充隐式反馈？

> **作为** 坐席，**我想要** 在客户情绪积极时收到合适的产品推荐，**以便** 适时进行交叉营销。

### US-9：系统降级自动容灾

> **作为** 系统，**我需要** 在 LLM/Embedding 服务不可用时自动降级，**以便** 客户和坐席仍能获得基本服务。

| 故障场景 | 降级行为 | 用户影响 |
|---------|---------|---------|
| LLM 熔断 (5 次连续失败) | 分类降级为 Rule only; 生成降级为检索摘要 | 回答质量下降但可交互 |
| LLM + 检索都不可用 | 生成降级为意图模板 (10 种意图模板) | 建议客户"输入转人工" |
| Embedding 熔断 | 检索降级为 BM25 Only | 检索精度下降但仍有结果 |
| Temporal 不可用 | Assist 降级为同步编排器 (4 路并行) | 无 OE 策略矩阵但仍有辅助推送 |
| E3 风控熔断 | 降级为 pass_with_audit_flag | 放行但标记待审, 事后审计 |
| E1 AI 熔断 | 降级为 fast_path 或 safe_fallback | 坐席看到兜底话术 |
| E2 营销熔断 | 降级为 skip_card | 营销卡片不展示 |
| 全局超时 (>5s) | 返回 timeout_partial | 坐席收到部分结果 |

**验收标准**:
- [ ] LLM 熔断器: 5 次连续失败打开, 60s 后半开探测, 2 次成功关闭
- [ ] Embedding 熔断器: 30s 探测间隔, 2 次成功恢复, 3 次失败打开
- [ ] 降级切换对用户透明 (不报错, 不中断)
- [ ] 降级恢复后自动升级 (DEGRADED → NORMAL)
- [ ] 所有降级事件记录日志

**待讨论**:
- 是否需要运营端可查看当前降级状态？
- 熔断器参数 (失败阈值/恢复超时) 是否需要运行时可调？
- 降级时是否需要通知坐席端？

### US-10：客户从 Bot 平滑转接坐席

> **作为** 客户，**我想要** 在需要时无缝从机器人对话切换到人工客服，**以便** 我的完整对话上下文能被人工客服继承。

| # | 角色 | 交互/动作 | 系统行为 |
|---|------|----------|---------|
| 1 | CU | 在 Bot 对话中 | 会话阶段 = BOT |
| 2 | SYS | 触发转人工 (L1/L2/L3/业务直转) | 会话阶段 → HANDOFF |
| 3 | SYS | — | star-connection 创建人工会话 → 返回 pollUrl |
| 4 | CU | 跳转到人工客服界面 | — |
| 5 | AG | 在 Workbench 接入 | star-connection 回调 /session/update → 会话阶段 → ASSIST |
| 6 | SYS | — | 坐席 WS 注册 → 收到 assist_ready |
| 7 | SYS | — | OE 编排启动: 话术+知识+告警+产品 推送到坐席 |
| 8 | AG/CU | 对话结束 | star-connection 回调 → 会话阶段 → ENDED |

**验收标准**:
- [ ] 转接后坐席可看到客户与 Bot 的历史对话
- [ ] 坐席接入后 1s 内收到首次辅助推送
- [ ] 会话阶段正确流转: BOT → HANDOFF → ASSIST → ENDED
- [ ] WS 连接生命周期与 ASSIST 阶段对齐

**待讨论**:
- 转接时是否需要向坐席推送"转接摘要"（客户意图/情绪/关键信息）？
- 客户是否可以在 ASSIST 阶段回退到 BOT 自助？
- 转接等待时间过长时是否需要排队/提示机制？

## 五、关键设计决策

| # | 决策 | 选择 | 原因 | 待验证 |
|---|------|------|------|--------|
| 1 | 异步消息队列 | Redis List + BRPOP | 轻量, 避免阻塞 HTTP, 与会话状态共享 Redis | 大规模时是否需要换 Kafka |
| 2 | 双通道分类 | Rule 快路 + LLM 慢路 | 平衡速度与精度, 规则覆盖高频场景 | 规则覆盖率是否足够 |
| 3 | OE 状态机 | Temporal Workflow | 可观测+可恢复+策略矩阵; 不可用同步降级 | Temporal 运维复杂度 |
| 4 | 营销压制 | D1 激活 → D2 suppress 2 轮 | 避免服务场景过度营销 | 2 轮阈值是否合理 |
| 5 | 反馈缓冲 | 3 秒延迟提交 + Undo | 坐席误操作可撤销, 减少脏数据 | 3s 是否太短 |
| 6 | CAS 乐观锁 | Redis Lua 脚本 | 多执行器并发写回状态, 避免丢失更新 | 高并发下冲突率 |
| 7 | PII 脱敏 | 全局仲裁器递归处理 | 确保所有输出经过脱敏 | 正则是否遗漏 |
| 8 | 双写索引 | ES + Milvus | BM25 和向量各有优势, RRF 融合互补 | 一致性保障 |
| 9 | 风控降级 | pass_with_audit_flag | 风控宁可漏放不误拦, 事后审计兜底 | 审计流程是否闭环 |
| 10 | 入库同步 | 单文档同步入库 | Sprint 2 实验阶段便于即时反馈 | 大文档需异步化 |

## 六、待讨论议题

### 高优先级

1. **L3 累积转人工阈值**: 当前 3 轮连续低置信度，是否需要更灵活的策略？
2. **star-connection 容错**: 转人工桥接失败时的重试/降级策略
3. **合规规则动态化**: 运营后台增删合规规则的能力
4. **反馈闭环**: 隐式反馈数据如何用于优化推荐质量

### 中优先级

5. **文档审核流程**: 知识文档上传后是否需要审核环节
6. **推送暂停**: 坐席端"暂停推送"按钮的需求
7. **降级状态可视化**: 运营/坐席端展示当前降级状态
8. **转接摘要**: 转人工时向坐席推送客户上下文摘要

### 低优先级

9. **入库异步化**: 大文档后台处理 + 进度查询
10. **文档增量更新**: 新版本只更新变更分块
11. **文档过期下架**: expiry_date 自动下架机制
12. **坐席显式反馈**: 补充主动评分机制
