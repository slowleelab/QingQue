双系统
设计决策说明

为什么取消 HANDOFF 阶段？ 从客户视角，"我要找人工"之后无论排队还是接通，都是"人工服务"。排队等待是人工服务的冷启动过程，不是独立阶段。银行考核的是"人工服务全流程 SLA"，从客户请求转人工到话后小结结束。

为什么 ASSIST 改名为 AGENT？ ASSIST 指的是"坐席辅助"（AI 给坐席推话术），这是技术能力不是业务阶段。人工对话始终存在，但 AI 辅助可能降级甚至不可用。AGENT 是业务事实。

为什么 BOT 只剩 BOT_ACTIVE？ 欢迎语是 BOT_ACTIVE 的第一轮对话；低置信度是 BOT_ACTIVE 的内部计数器；请求转人工是事件不是状态；空闲超时是 BOT_ACTIVE 的事件。系统行为在这些场景下完全一致，只是内部计数器不同。

为什么 AG_QUEUED 和 AG_ASSIGNED 分开？ 行业标准（Amazon Connect、Genesys、Twilio）都分开。系统行为不同（等待音 vs 接通提示）、超时含义不同（无坐席 vs 坐席未接听）、SLA 指标不同（排队时长 vs 振铃时长）。

为什么取消 AG_TRANSFERRING？ 坐席转接的本质：释放旧坐席 → 分配新坐席 → 等待接受。这就是 AG_ACTIVE → AG_ASSIGNED，转接是动作不是状态。

为什么 ENDED 没有子状态？ ENDED 是终态，不会再有转换。END_COMPLETED / END_TIMEOUT 等是结束原因（end_reason 字段），不是状态。

方案

SessionPhase 扩展方案

方案：Phase + SubPhase 两层结构
SessionPhase 3 个值 (BOT / AGENT / ENDED)，新增 SessionSubPhase 枚举承载子状态。
SessionState.current_phase + SessionState.current_sub_phase 组合决定完整状态。
转换规则：transition_phase(new_phase, sub_phase=None, event=None)，内部验证白名单。
ENDED 的结束原因记录在 SessionState.end_reason 字段，不作为子状态。
SessionPhase (3 值)
class SessionPhase(StrEnum):
    BOT    = "bot"
    AGENT  = "agent"   # 替代 HANDOFF+ASSIST
    ENDED  = "ended"
SessionSubPhase (7 值)
class SessionSubPhase(StrEnum):
    # BOT
    BOT_ACTIVE   = "bot:active"
    # AGENT
    AG_QUEUED    = "agent:queued"
    AG_ASSIGNED  = "agent:assigned"
    AG_ACTIVE    = "agent:active"
    AG_ON_HOLD   = "agent:on_hold"
    AG_REVIEWING = "agent:reviewing"
    # ENDED (无子状态，用 end_reason)
合法转换白名单
VALID_TRANSITIONS = {
