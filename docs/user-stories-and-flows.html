转人工三步：发起（SmartCS 调用 star-conn）→ 等待（star-conn 内部路由，SmartCS 不感知）→ 激活（坐席接听，Redis 通知 SmartCS）。
star-conn 内部的路由细节对 SmartCS 是黑盒。SmartCS 只关心两件事：转了没、接了没。

五、坐席 UI 与辅助系统交互
1 个长连接 (WS 推送) + 多个短请求 (REST 控制)。WS 负责"辅助内容下行"，REST 负责"会话状态上行"。
star-conn ↔ Assist 通信架构
通知面 (异步): POST /notify — star-conn 发送通知后立即返回 202 (<10ms)，Assist 内部 asyncio.Queue 异步消费 + 编排 + WS 推送。不阻塞 star-conn 线程。
控制面 (同步): POST /session/update /hold /resume /review/* /feedback — 低频请求-响应，HTTP 完全合适。
过载处理: Queue 满 → 503 → star-conn 感知并降级。Assist 宕机 → 坐席仍可正常对话，无辅助推送。
为什么不用 Kafka/Redis Streams: Assist 宕机时积压消息无恢复价值；过载时 503 比 Stream 缓冲更诚实；零新增中间件。
🔌 WebSocket 连接
· 建连: ws://assist:8001/api/ws/agent/{agent_id}
· 就绪消息: {"type":"connected"}
· 心跳: 15s 间隔, 30s 超时断开
· 注册到 ws_pool (session_id → WebSocket)
📨 消息触发入口
· 主路径: star-connection → POST /notify (202 异步)
· 备用: 前端 WS 发送 {"type":"customer_message"}
· 编排: 优先 Temporal, 降级 AssistOrchestrator
· 推送: {"type":"assist_push", "payload":{...}}
⏸ Hold / Resume
· 保持: POST /hold → AG_ACTIVE → AG_ON_HOLD
· 静音检测: 60s 无客户消息 → WS 推送 silence_alert
· 恢复: POST /resume → AG_ON_HOLD → AG_ACTIVE
· 恢复后静音检测自动取消
📝 Review 话后小结
· 生成: POST /review/generate → LLM 小结
· 推送: WS {"type":"call_summary"}
· 提交: POST /review/submit → ENDED
· 降级: LLM 不可用时用模板小结
💬 Feedback 隐式反馈
· 采纳: POST /feedback action=accept conf=1.0
· 修改: action=modify conf=0.5
· 部分采纳: action=partial_accept conf=0.3
· 撤销: POST /feedback/undo (3s 内)
⚡ 节流机制
· 同一会话 throttle_window_ms 内只推送一次
· critical 级别告警不受节流限制
· force_reset_throttle: 告警后重置节流
· 确保紧急信息不被延迟
WS 推送事件汇总
事件类型触发条件方向
assist_push每次客户消息分析完成服务端 → 坐席UI
call_summary话后小结生成完成服务端 → 坐席UI
session_timeout超时守卫触发 (排队/振铃/会话/话后超时)服务端 → 坐席UI
silence_alertON_HOLD 60秒无客户消息服务端 → 坐席UI
session_ended会话 ENDED服务端 → 坐席UI (同时关闭WS)
heartbeat每15秒服务端 → 坐席UI (保活)

  • 坐席接听后 3-8s 收到首次辅助推送 (LLM+RAG 编排)
  • 会话阶段正确流转: BOT → AG_QUEUED → AG_ASSIGNED → AG_ACTIVE → AG_REVIEWING → ENDED

七、Bot 与 Assist 内部运行机制与技术栈

7.1 Bot (:8000) 内部架构

HTTP 线程: POST /chat/send → XADD chat:stream → 202 | GET /chat/poll → GET response:{sid} (快路) / SUBSCRIBE notify:{sid} (慢路)
后台协程 (_consumer_loop): XREADGROUP COUNT 10 BLOCK 1s → create_task(_process_message) 秒返
后台协程 (_process_message): 幂等→per-session Queue→Semaphore(有空槽?Agent:fast_reply)→SETEX→PUBLISH→XACK
后台协程 (_claim_stale): XAUTOCLAIM min_idle=60s → 认领挂死消息
组件技术用途
Web 框架FastAPI + uvicornHTTP 路由, 长轮询
消息通道Redis Streams (XADD/XREADGROUP/XACK)消息持久化, at-least-once
宕机恢复XAUTOCLAIM超时未确认消息自动认领
异步编排asyncio (create_task / Semaphore / per-session Queue)并发消费, Queue 保序, 过载保护
通知机制Redis Pub/Sub (PUBLISH/SUBSCRIBE)轮询即时唤醒
AI 编排LangGraph StateGraphSupervisor 路由
意图分类Rule(regex) + LLM(Qwen2.5)双通道,快路<10ms,慢路~200ms
熔断降级CircuitBreakerLLM/Embedding不可用自动降级

7.2 Assist (:8001) 内部架构

WS 线程: WS /ws/agent/{agent_id} 坐席登录建连 → ws_pool[agent_id] = WebSocket, 生命周期=上班周期
REST 线程: POST /notify (star-conn→202→asyncio.Queue) | POST /hold|/resume | POST /review/* | POST /feedback
后台协程 (_notify_consumer): Queue.get → create_task(_handle_notify) 秒返
后台协程 (_handle_notify): 幂等→per-session Queue→get_session(Redis)→意图分类→OE编排(Temporal主/同步降级)→WS推送→合规检测→反馈推断
兜底协程: _timeout_guard (per sub-phase) | _silence_detector (60s ON_HOLD)
组件技术用途
Web 框架FastAPI + uvicornHTTP + WebSocket
消息入口HTTP POST /notify → asyncio.Queuestar-conn 异步通知
异步编排asyncio (create_task / Queue / per-session Worker)并发消费, Worker 保序
OE 编排Temporal (主) / AssistOrchestrator (降级)D评估→策略矩阵→E执行→仲裁
WS 管理ws_pool: dict[agent_id, WebSocket]按坐席索引, 一条持久WS
合规检测AlertEngine (10条银行规则)BLOCK/WARN/PASS, <100ms
超时守卫asyncio.Task per sub-phase6种超时, 阶段切换自动重建

八、关键设计决策
#决策选择原因
1会话状态模型3 Phase × 7 SubPhase层级清晰，子阶段驱动超时/推送策略
2Bot 消息通道Redis Streams + Consumer Group + PEL持久化 at-least-once, XACK, XAUTOCLAIM
3双通道分类Rule 快路 + LLM 慢路平衡速度与精度
4转人工三级触发L1 > L2 > L3优先级明确，高级别直接触发
5OE 状态机Temporal + 同步降级可观测+可恢复+策略矩阵
6坐席交互WS 推送 + REST 控制推送走长连接，控制走短请求，职责分离
7营销压制D1 激活 → D2 suppress 2 轮避免服务场景过度营销
8反馈缓冲3 秒延迟提交 + Undo坐席误操作可撤销
9CAS 乐观锁Redis Lua 脚本多执行器并发写回, 避免丢失更新
10PII 脱敏全局仲裁器递归处理确保所有输出经过脱敏
11超时守卫asyncio.Task per sub-phase细粒度超时，阶段切换自动重建
12风控降级pass_with_audit_flag风控宁可漏放不误拦, 事后审计兜底
13star-conn→Assist 通信HTTP /notify (202 异步)不阻塞 star-conn, 零新增中间件, 过载 503 比 Stream 缓冲更诚实
14Bot 过载保护Semaphore(10) + 固定话术兜底满荷不走 Agent，<50ms 固定话术，不拒客、不堆积
九、待讨论议题
高优先级
  1. L3 累积转人工阈值: 当前 3 轮连续低置信度，是否需要更灵活的策略？
  2. star-connection 容错: 转人工桥接失败时的重试/降级策略
  3. 合规规则动态化: 运营后台增删合规规则的能力
  4. 反馈闭环: 隐式反馈数据如何用于优化推荐质量
中优先级
  1. 文档审核流程: 知识文档上传后是否需要审核环节
  2. 推送暂停: 坐席端"暂停推送"按钮的需求
  3. 降级状态可视化: 运营/坐席端展示当前降级状态
  4. 转接摘要: 转人工时向坐席推送客户上下文摘要
  5. WS 断连恢复: WebSocket 意外断开后是否需要自动重连+消息补发
  6. 排队超时回退: 排队超时回退 BOT 后是否需要通知客户
低优先级
  1. 入库异步化: 大文档后台处理 + 进度查询
  2. 文档增量更新: 新版本只更新变更分块
  3. 坐席显式反馈: 补充主动评分机制
