SmartCS Bot 架构文档 v2.3

银行信用卡智能客服平台 — 交互时序图 · 设计决策 · 最佳实践

0. 系统概览

HTTP 消息入口

POST /api/chat/send → Redis Streams XADD → 202 Accepted。客户端不等待结果，通过 GET /api/chat/poll 长轮询获取回复。

Worker 消息消费

XREADGROUP Consumer Group 批量消费 → per-session Queue + 独享协程串行处理。XAUTOCLAIM 每 30s 认领挂死消息。

Agent 编排引擎

LangGraph 有向图：classify → supervisor → {knowledge | business | fallback} → transfer_check → respond/transfer。

LLM 推理

Qwen2.5:7B via Ollama，OpenAI 兼容 API。json_mode 结构化输出，熔断器 + 降级链保护。

检索 混合检索引擎

ES BM25 ∥ Milvus 向量 ANN → RRF 融合 → 可选 Reranker 精排。Redis 缓存 300s TTL。

审计 合规落库

chat_message 表全量记录，PG ACID 保证。tsvector GIN 全文索引。合并消息标记 source="merged"。

1. 主链路：从 HTTP 到 LLM 回复的 22 步

时序图 — 全链路消息处理

处理流程说明

Phase 1 — HTTP 同步 (<1ms)：客户端 POST 消息到 /api/chat/send，服务端做输入校验后 XADD 写入 Redis Stream 并立即返回 202。此时消息已持久化，客户端通过 GET /api/chat/poll 轮询获取结果。这种"写后即返"模式确保了 HTTP 请求不阻塞，即使后端 LLM 推理需要 5-10 秒。

Phase 2 — Worker 异步 (~5-10s)：后台 Consumer Loop 通过 XREADGROUP 批量拉取消息，按 session_id 路由到 per-session Queue。每个会话拥有一个独享协程 (Session Worker)，串行消费消息，无需加锁。Worker 经过过期检查、幂等检查、Semaphore 过载保护后，调用 Agent 编排引擎。

Phase 3 — 回复交付：Agent 生成回复后，Worker 将结果写入 Redis response key（TTL 120s）并通过 Pub/Sub 通知客户端。客户端的长轮询即刻被唤醒，获取结果。同时审计记录更新为 status=done。

设计决策：消息通道选择
方案考虑结论
直接 HTTP 同步LLM 推理 5-10s → HTTP 超时风险，客户端阻塞❌ 不可接受
Kafka适合事件溯源和跨服务扇出，但 Bot 是请求-响应模式，引入 Kafka 增加运维复杂度❌ 已从代码中移除
Redis Streams + Consumer Group低延迟、at-least-once 保证、XAUTOCLAIM 挂死恢复、无需额外中间件✅ 采用
Per-session Queue + Worker串行保序无需锁、空闲 300s 自动退出、独占协程无竞态✅ 采用

方法论：请求-响应模式的实时系统优先选择 Redis Streams 而非 Kafka。Kafka 适用于事件溯源和跨服务异步解耦，不适合需要即时回复的同步等待场景。

2. 意图分类：双通道 Fast/Slow 路径

时序图 — Fast Path 正则 + Slow Path LLM 两级降级

双通道原理

Fast Path (RuleClassifier)：纯内存正则 + 关键词匹配，<1ms 延迟。10 条预定义规则覆盖高频场景（账单、额度、分期、积分、挂失、投诉、转人工、闲聊）。命中的规则输出固定置信度。

Slow Path (LLMClassifier)：当 Fast Path 置信度 < 0.7 时触发。Qwen2.5:7B 通过 json_mode 输出结构化结果 {intent, confidence, entities, sentiment}。LLM 置信度 < 0.3 时标记 source="fallback"，表示分类不可靠。

熔断保护：LLM 熔断器打开时跳过 Slow Path，直接用 Fast Path 的低置信度结果兜底，确保系统在 LLM 故障时仍可用。

设计决策：为什么不用单一 LLM 分类
方案延迟准确率可用性结论
纯正则规则<1ms60-70%（高频场景高，长尾低）100%（无外部依赖）❌ 长尾覆盖不足
纯 LLM 分类1-5s85-90%依赖 Ollama 可用❌ 延迟不可接受
双通道 Fast+Slow<1ms (60%) / 1-5s (40%)85-90%LLM 不可用时降级到 Fast Path✅ 最优折衷

最佳实践：实时系统中，绝大多数请求（60-80%）应通过低成本路径处理。Only ambiguous cases go to the expensive path. 这是 "cheap-first, expensive-fallback" 模式。

3. 混合检索：ES BM25 ∥ Milvus ANN → RRF 融合

时序图 — knowledge_agent 内部检索流程（并行 + 融合）

为什么需要两路检索

BM25 和向量检索解决不同的问题。BM25 基于词频-逆文档频率，擅长精确关键词匹配，例如"白金卡年费减免"会精确命中包含这些词的文档。但它无法理解"怎么免年费"和"年费减免条件"是同一个意图。

向量检索 基于语义相似度，将查询和文档映射到 1024 维向量空间，余弦距离衡量语义相关性。询问"怎么才能不交年费"时，即使知识库里没有"不交年费"这个词，向量检索也能找到"年费减免条件"相关文档。

RRF 融合：BM25 分数范围 0-20，Cosine 分数范围 0-1，两种分数无法直接比较。RRF 按排名融合，公式 score = Σ 1/(k + rank_i)，不依赖分数绝对值，天然免疫量纲差异。

设计决策：hybrid search 为什么优于 single-source
方案关键词搜索语义搜索故障容错结论
纯 ES BM25✅ 精确❌ 无法理解同义词/改写单点故障❌
纯 Milvus 向量❌ 无法精确匹配✅ 语义理解单点故障❌
Hybrid + RRF✅ BM25✅ 向量 ANN✅ 一路失败自动降级✅

最佳实践：生产级 RAG 系统必须使用混合检索。单一检索引擎在特定场景下会失效，且没有降级路径。RRF 是业界标准融合算法（Google、Elastic、Weaviate 均采用），不需要调权重，不需要归一化。

4. 队列合并：Phase 2 调用前检查

时序图 — LLM 调用前 peek 队列，drain 后合并为一次调用

工作原理

正常单条消息场景（约 99% 请求），Worker 在 classify+RAG 完成后 peek q.empty() 发现队列为空，直接调用 LLM。整个过程 零额外延迟。

连续消息场景（约 1% 请求），Worker 在 LLM 调用前发现队列中有排队消息，立即 q.get_nowait() 全部 drain 出来。跳过过期消息，为有效消息写入审计记录，然后将所有消息合并为一条 "A\nB\nC"，只调用一次 LLM。

关键设计：不是"等待窗口"（每条消息等 300ms），而是"取消检查点"——消息立即开始处理，只在最贵的 LLM 调用前检查是否合并。这是零延迟的关建。

设计决策：为什么不用前置积累门 (300ms wait)
方案单消息延迟连续消息优化复杂度结论
前置积累门 (300ms wait)+300ms (每条)有效低❌ 99% 请求白等
调用前 peek 队列0有效 (省 N-1 次 LLM)中✅ 零惩罚
不处理 (status quo)0无 (3 条 → 3 次 LLM)无❌ 浪费算力

最佳实践：不做"等待"机制，做"检查点"机制。在昂贵的操作前做一个零成本的检查（peek 队列是纯内存操作），而非在开始处理前人为加延迟。这是 Google Tail Latency 优化中的 "do not wait, check before commit" 原则。

5. 转人工判断：L1L2L3 三级触发

时序图 — 从 agent 输出到 respond/transfer 分支

三级触发机制

L1 (关键词触发，<1ms)：用户在输入中明确说了"转人工"、"找真人"、"人工客服"等关键词。这是最明确的转人工信号，直接触发。

L2 (语义触发，分类阶段已完成)：LLM 分类时输出的 sentiment 字段为 angry（愤怒），且置信度 > 0.8。表示客户情绪极度负面，即使没有明确说"转人工"，也主动触发以避免事态升级。

L3 (累计触发)：连续 3 轮 primary_confidence < 0.5，说明 Bot 持续无法理解客户意图。此计数器存储在 Redis Session 中，每轮更新，防止 Bot 陷入"不懂→瞎猜→更不懂"的循环。

设计决策：三级 vs 单一规则
方案覆盖误触发延迟结论
仅关键词 (L1)低（仅显式请求）无<1ms❌ 漏掉情绪和低置信度场景
仅语义 (L2)中（情绪检测）中（假阳性）随分类完成❌ 无法处理关键词触发
L1+L2+L3 分层高低（L1 最可靠优先，L3 累计确认）O(1)✅ 逐层降级

最佳实践：多级触发机制遵循 "最可靠信号优先" 原则。L1 精确关键词 → 零误触发；L2 语义情绪 → 快速响应；L3 累计低置信度 → 兜底保障。每级都是上一级未命中时的 fallback，而非并行判断。

6. 保护机制：熔断 + 降级 + 过载 + 审计

系统保护全景

SmartCS Bot 部署了多层保护机制，确保在外部依赖故障、流量尖峰、资源耗尽等场景下仍能对外提供服务（可能降级，但不会崩溃或拒客）。

LLM 降级链 DegradationManager HealthMonitor 指数退避探测 LLM 不可用 NORMAL → DEGRADED(跳过 LLM 用检索摘要) → FALLBACK(跳过检索用模板)

设计决策：银行客服的可用性要求

核心原则：永远不拒客。银行客服不同于互联网产品，客户发出消息后必须得到回复，即使回复是"当前繁忙请稍候"也不能无声无息。这决定了所有保护机制的基调——降级而非拒绝。

场景普通系统做法SmartCS 做法
LLM 超时/不可用返回 503 或内部错误跳过 LLM → 检索摘要 → 模板回复。source 标记降级来源
并发满载拒绝请求Semaphore 满荷 → Fast Reply 固定话术。不排队、不拒绝
ES/Milvus 故障搜索失败一路失败自动切换另一路 → 两路全失败则空 context → LLM 直接生成
消息积压堆积至超时TTL 8s 过期跳过 + XAUTOCLAIM 60s 挂死恢复

最佳实践：SRE 三原则 — (1) 每层独立降级，不做级联故障传播；(2) 熔断器采用连续失败计数而非固定时间窗口（防止间歇性故障反复触发）；(3) 降级路径不依赖 LLM（确保在 LLM 自身故障时降级路径仍然可用）。
