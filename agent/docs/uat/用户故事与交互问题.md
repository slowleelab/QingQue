# SmartCS 交互问题梳理与用户故事

> **日期**: 2026-05-13
> **范围**: 全链路交互：客户 → Bot → 转人工 → 坐席辅助 → 会话结束

---

## 一、交互问题全景

### 1.1 问题清单

| # | 问题 | 严重性 | 影响范围 | 根因 |
|---|------|--------|----------|------|
| P1 | 转接时对话历史永远为空，坐席"盲接" | **P0** | 坐席体验 | `build_transfer_request()` 传 `history=[]`，从未提取 Redis 中的对话记录 |
| P2 | star-connection 无 TransferController，`POST /api/sessions` 不处理转接请求 | **P0** | 转接流程 | Java 端仅有 MessageController，无 TransferController 接收 Bot 传来的上下文 |
| P3 | star-connection 不回调 `session/update`，会话阶段永远停在 HANDOFF | **P0** | 状态同步 | Java SessionManager 无 HTTP 回调到 Assist 服务 |
| P4 | 客户端无等待/排队 UI，转接时直接显示"已连接" | **P1** | 客户体验 | chatStore 收到 `is_transfer` 后直接设 `agentConnected=true`，无中间态 |
| P5 | 反馈闭环前端未实现（accept/modify/reject 无按钮） | **P1** | 数据闭环 | AssistPanel 的 ScriptCard 无操作按钮，/feedback API 无前端调用方 |
| P6 | 会话结束无感知，客户可继续输入到死会话 | **P1** | 客户体验 | star-connection 的 poll 不返回 `session_ended`，前端无终止检测 |
| P7 | 两套状态机不同步（Redis SessionPhase vs Java SessionStatus） | **P1** | 一致性 | HANDOFF/ASSIST/ENDED 与 WAITING/ACTIVE/CLOSED 无映射同步机制 |
| P8 | 客户消息未存为 DialogueTurn，历史只有 Bot 回复 | **P1** | 上下文 | LangGraph 只在 respond_node/handoff_node 记录 Bot 回复，不记录客户输入 |
| P9 | WebSocket 连接池进程内存储，多 worker 下推送丢失 | **P2** | 消息可靠 | `app.state.assist_ws_connections` 是普通 dict，跨进程不可见 |
| P10 | 坐席端消息无去重，可能重复显示 | **P2** | 体验 | ConversationPanel.pollMessages 无 messageId 去重 |

## 二、用户故事

按 4 个阶段组织，每个故事包含：场景描述、验收标准、涉及问题编号。

---

### 阶段1: BOT 自助对话

#### US-01 客户发起首条消息

**作为** 客户，**我希望** 在网页端输入消息后立即看到 Bot 回复，**以便** 快速获得我需要的信息。

**场景**:
1. 客户打开聊天页面
2. 输入"你好"，点击发送
3. 系统显示"正在输入..."
4. Bot 返回问候语

**验收标准**:
- [ ] 消息发送后 3 秒内出现"正在输入..."提示
- [ ] Bot 在 5 秒内返回回复
- [ ] 回复内容为合理的问候语
- [ ] 界面自动生成 session_id，后续消息复用

**涉及问题**: 无（当前流程正常）

---

#### US-02 客户查询业务问题

**作为** 客户，**我希望** 询问信用卡相关问题时 Bot 能给出准确回答，**以便** 不需要转人工。

**场景**:
1. 客户输入"年费怎么收"
2. Bot 意图识别为 bill_query
3. RAG 检索知识库，返回年费政策

**验收标准**:
- [ ] Bot 正确识别意图为 bill_query
- [ ] 回复来源标注为 RAG（source="rag"）
- [ ] 回复内容包含具体年费金额和减免条件
- [ ] 客户消息和 Bot 回复均完整保存在会话历史中

**涉及问题**: P8（客户消息未存为 DialogueTurn）

---

#### US-03 连续多轮对话

**作为** 客户，**我希望** 与 Bot 进行连续对话时它能记住上下文，**以便** 不需要重复描述问题。

**场景**:
1. 客户: "查询账单" → Bot 返回账单信息
2. 客户: "怎么还款" → Bot 理解是同一会话的后续问题
3. 客户: "谢谢" → Bot 识别对话结束

**验收标准**:
- [ ] 三轮对话使用同一 session_id
- [ ] 每轮对话的意图正确识别
- [ ] 会话历史中包含完整的对话记录（含客户输入）
- [ ] 对话上下文在多轮间保持

**涉及问题**: P8

---

### 阶段2: 转人工（Handoff）

#### US-04 客户主动要求转人工

**作为** 客户，**我希望** 输入"转人工"后看到等待提示并被转接到真人坐席，**以便** 解决 Bot 无法处理的问题。

**场景**:
1. 客户输入"我要转人工"
2. Bot 识别意图为 transfer_agent，触发 L1 关键词
3. 界面显示"正在为您转接人工坐席，请稍候..."
4. 系统将对话历史和上下文传递给 star-connection
5. 坐席接起后，界面显示"坐席已接入"

**验收标准**:
- [ ] 触发转人工后，客户立即看到等待提示（非直接"已连接"）
- [ ] 等待期间有动态指示（如"排队中，前方N人"或"正在匹配坐席"）
- [ ] 坐席接入后，客户收到明确通知
- [ ] 转接传递完整对话历史（非空列表）
- [ ] 转接传递意图、情绪等上下文信息
- [ ] 若坐席系统不可用，客户看到友好的错误提示

**涉及问题**: P1, P2, P4

---

#### US-05 坐席收到带上下文的转接

**作为** 坐席，**我希望** 接到转接会话时能看到客户之前的对话历史、意图和情绪，**以便** 快速理解客户需求。

**场景**:
1. 客户与 Bot 对话3轮后触发转人工
2. 坐席工作台出现新会话通知
3. 坐席点击接起，看到完整的对话摘要
4. 对话摘要包括: 客户原始问题、Bot 回复、意图标签、情绪标签

**验收标准**:
- [ ] 坐席接起会话后能看到客户与 Bot 的完整对话历史
- [ ] 历史记录标注每轮的意图和置信度
- [ ] 显示客户的情绪标签（如 positive/neutral/negative/angry）
- [ ] 显示转接原因（如"客户主动请求"/"情绪升级"/"挂失业务"）
- [ ] 无历史为空的情况

**涉及问题**: P1, P2

---

#### US-06 坐席接起后会话阶段同步

**作为** 系统，**我希望** 坐席接起会话后，SmartCS 和 star-connection 的会话状态保持同步，**以便** 后续的辅助推送和反馈闭环正常工作。

**场景**:
1. 客户被转接，SessionPhase = HANDOFF
2. star-connection 分配坐席，SessionStatus = ACTIVE
3. star-connection 回调 SmartCS POST /api/session/update(ASSIST)
4. SmartCS 更新 SessionPhase = ASSIST
5. 客户和坐席界面同步显示"坐席服务中"

**验收标准**:
- [ ] 坐席接起后，SmartCS SessionPhase 从 HANDOFF 变为 ASSIST
- [ ] star-connection SessionStatus 为 ACTIVE
- [ ] 两套状态机映射一致: HANDOFF↔WAITING, ASSIST↔ACTIVE, ENDED↔CLOSED
- [ ] 客户端界面反映当前阶段
- [ ] 坐席端界面反映当前阶段

**涉及问题**: P3, P7

---

#### US-07 转接超时处理

**作为** 客户，**我希望** 如果长时间没有坐席接起，系统能给出反馈而不是无限等待，**以便** 我可以选择继续等或放弃。

**场景**:
1. 客户触发转人工
2. 60秒内无坐席接起
3. 系统提示"当前坐席繁忙，是否继续等待？"
4. 客户选择"继续等待"或"返回机器人对话"

**验收标准**:
- [ ] 等待超过60秒后显示超时提示
- [ ] 提供"继续等待"和"返回Bot"两个选项
- [ ] 选择"返回Bot"后恢复Bot对话
- [ ] 超时不会导致会话卡死在 HANDOFF 阶段

**涉及问题**: P4, P6（部分，需要 HANDOFF 超时回退机制）

---

### 阶段3: 坐席辅助（Assist）

#### US-08 坐席收到实时辅助推送

**作为** 坐席，**我希望** 客户发送消息后我能实时看到 AI 推荐的话术、知识片段和风险告警，**以便** 更专业地服务客户。

**场景**:
1. 坐席已接起会话
2. 客户发送"我上个月的退款怎么还没到"
3. star-connection 回调 /api/analyze
4. 坐席右侧面板实时更新:
   - 推荐话术: "关于退款进度，请您..."
   - 知识片段: 退款流程FAQ
   - 风险告警: 无

**验收标准**:
- [ ] 客户消息发送后 2 秒内坐席看到辅助推送
- [ ] 推送内容包括: 话术卡片 + 知识片段 + 告警
- [ ] 话术内容与客户问题相关
- [ ] 推送通过 WebSocket 实时到达（非轮询）
- [ ] 无重复推送

**涉及问题**: P9（多 worker 下 WS 推送丢失）

---

#### US-09 坐席采纳/修改/拒绝 AI 建议

**作为** 坐席，**我希望** 能对 AI 推荐的话术进行采纳、修改或拒绝操作，**以便** 保持我的专业判断同时利用 AI 辅助。

**场景**:
1. 坐席看到推荐话术: "关于退款进度..."
2a. 坐席直接点击"采纳" → 话术自动填入输入框 → 发送
2b. 坐席修改话术后点击"修改发送" → 标记修改了哪些字段
2c. 坐席忽略话术，自行输入回复 → 标记为"拒绝"

**验收标准**:
- [ ] 每张话术卡片有"采纳""修改""忽略"三个操作按钮
- [ ] 采纳: 话术填入输入框，confidence=1.0
- [ ] 修改: 可编辑话术内容后发送，记录 modify_fields
- [ ] 忽略: 不操作，3秒后自动标记 confidence=0.0
- [ ] 3秒内可撤销操作
- [ ] 反馈数据写入 Redis（延迟提交机制）

**涉及问题**: P5

---

#### US-10 风控拦截场景

**作为** 坐席，**我希望** 当客户提出违规请求时（如套现），AI 立即发出醒目的风控拦截告警，**以便** 避免违规操作。

**场景**:
1. 客户发送"你们能帮我套现吗"
2. AI 检测到 R-COMP-001 规则命中（critical）
3. 坐席右侧面板顶部出现红色风控拦截卡片
4. 营销推荐被完全隐藏
5. 话术推荐切换为合规话术

**验收标准**:
- [ ] 风控告警在 1 秒内显示
- [ ] 告警级别为 critical，使用红色醒目样式
- [ ] 主卡片被替换为风控拦截卡片（fusion_type=risk_blocked）
- [ ] 营销卡片不显示
- [ ] 告警包含具体的违规类型和建议操作

**涉及问题**: 无（后端已实现，前端需验证渲染）

---

#### US-11 情绪告警与趋势

**作为** 坐席，**我希望** 当客户情绪恶化时收到告警提示，**以便** 及时调整沟通策略或申请升级处理。

**场景**:
1. 客户连续3轮表达不满
2. AI 检测到情绪趋势: 连续3轮 negative/angry
3. 坐席面板出现"客户情绪不佳"的黄色告警
4. 告警建议: "表达理解和同理心，积极解决问题"

**验收标准**:
- [ ] 客户 angry 情绪触发 critical 告警
- [ ] 客户 negative 情绪触发 warning 告警
- [ ] 连续3轮负面情绪触发趋势告警
- [ ] 告警包含安抚话术建议
- [ ] 趋势告警级别高于单轮情绪告警

**涉及问题**: 无

---

### 阶段4: 会话结束（Ended）

#### US-12 坐席结束会话

**作为** 坐席，**我希望** 通话结束后点击"结束"按钮，系统自动通知客户并生成通话摘要，**以便** 规范地结束服务。

**场景**:
1. 坐席点击"结束通话"按钮
2. star-connection 关闭会话
3. star-connection 回调 SmartCS POST /api/session/update(ENDED)
4. 客户端收到会话结束通知
5. 客户端显示满意度评价页面

**验收标准**:
- [ ] 坐席点击结束按钮后，会话状态变为 ENDED
- [ ] star-connection 回调 SmartCS 更新 SessionPhase
- [ ] 客户端收到 session_ended 信号
- [ ] 客户端停止轮询，显示"通话已结束"
- [ ] 客户端展示满意度评价入口
- [ ] 结束后客户无法继续发送消息

**涉及问题**: P3, P6

---

#### US-13 会话状态一致性

**作为** 运维人员，**我希望** SmartCS 和 star-connection 的会话状态始终保持一致，**以便** 不出现状态漂移导致的故障。

**场景**:
1. BOT 阶段: SmartCS=BOT, star-conn=N/A
2. HANDOFF 阶段: SmartCS=HANDOFF, star-conn=WAITING
3. ASSIST 阶段: SmartCS=ASSIST, star-conn=ACTIVE
4. ENDED 阶段: SmartCS=ENDED, star-conn=CLOSED

**验收标准**:
- [ ] 每个阶段两套系统的状态映射正确
- [ ] 阶段转换由明确的回调驱动，非前端猜测
- [ ] 任一系统不可用时，另一系统有超时回退机制
- [ ] 可通过 API 查询两套系统的状态并验证一致性

**涉及问题**: P3, P7

---

#### US-14 通话摘要

**作为** 坐席，**我希望** 通话结束后能看到 AI 自动生成的通话摘要，**以便** 快速归档和交接。

**场景**:
1. 通话结束
2. AI 基于对话历史生成摘要: 客户需求、处理结果、情绪状态
3. 摘要显示在坐席工作台

**验收标准**:
- [ ] 摘要包含: 客户需求、问题分类、解决方案、解决状态
- [ ] 摘要包含客户情绪标签和置信度
- [ ] 摘要在通话结束后 5 秒内生成
- [ ] 坐席可以编辑摘要

**涉及问题**: 无（后端 CallSummary 模型已定义，需实现生成逻辑和前端展示）

### 跨阶段用户故事

#### US-15 PII 脱敏全链路

**作为** 客户，**我希望** 我的个人信息（手机号、银行卡号、身份证号、姓名）在坐席端显示时自动脱敏，**以便** 保护我的隐私。

**场景**:
1. 客户发送"我的手机号是13800138000"
2. AI 生成话术中包含该手机号
3. 坐席看到的是"手机号是[PHONE]"
4. 同理银行卡号、身份证号、姓名均被脱敏

**验收标准**:
- [ ] 手机号 (1[3-9]开头的11位) → [PHONE]
- [ ] 银行卡号 (16-19位) → [BANKCARD]
- [ ] 身份证号 (18位) → [IDCARD]
- [ ] 客户姓名 → [NAME]（在"客户XX的"等上下文中）
- [ ] 所有卡片（话术、风险、营销）均脱敏
- [ ] 短数字（如"3位"）不被误脱敏

**涉及问题**: 无（后端已实现）

---

#### US-16 降级不影响核心对话

**作为** 客户/坐席，**我希望** 即使部分 AI 服务不可用，对话仍能正常进行，**以便** 业务不中断。

**场景**:
1. LLM 服务宕机 → Bot 降级为模板回复，坐席辅助降级为安全话术
2. ES 宕机 → 检索降级为向量检索
3. Milvus 宕机 → 检索降级为 BM25
4. ES+Milvus 宕机 → 检索空结果，Bot 用模板回复
5. Temporal 宕机 → 辅助降级到同步编排器

**验收标准**:
- [ ] 每种降级场景下对话不中断
- [ ] 降级结果有明确标记（degraded=true, degradation_type）
- [ ] 服务恢复后自动恢复正常模式
- [ ] 降级事件记入日志/监控

**涉及问题**: 无（后端已实现降级链）

## 三、修复优先级建议

| 优先级 | 修复项 | 涉及故事 | 工作量估算 |
|--------|--------|----------|-----------|
| **P0-1** | 补全 TransferController (Java) + 传递对话历史 | US-04, US-05 | 3天 |
| **P0-2** | star-connection 回调 session/update | US-06, US-12 | 2天 |
| **P0-3** | 客户消息存入 DialogueTurn | US-02, US-03 | 0.5天 |
| **P1-1** | 客户端等待/排队 UI | US-04 | 1天 |
| **P1-2** | 前端反馈按钮 (accept/modify/reject) | US-09 | 2天 |
| **P1-3** | 会话结束通知 + 评价页 | US-12 | 1.5天 |
| **P1-4** | 状态机同步机制 | US-13 | 1天 |
| **P2-1** | WS 连接池 Redis 化 | US-08 | 2天 |
| **P2-2** | 坐席端消息去重 | US-08 | 0.5天 |

**建议修复顺序**: P0-3 → P0-1 → P0-2 → P1-1 → P1-2 → P1-3 → P1-4 → P2-1 → P2-2

先修最小代价的 P0-3（客户消息存储），再修最关键的 P0-1（转接上下文传递），再修 P0-2（状态同步回调），然后逐步补全前端体验。
